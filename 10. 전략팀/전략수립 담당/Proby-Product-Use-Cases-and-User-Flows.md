# Proby 제품 Use Case & User Flow 시나리오

> Proby Strategy Ver.2 + 전략대화(에이전트 시대·Proby Moat)가 반영된 제품의 모습, 사용 시나리오, 사용자 플로우.

---

## 1. 제품이 지향하는 모습 (전략·전략대화 반영)

### 1.1 한 줄 정의

Proby는 **“한 번 쓰고 끝나는 리서치 툴”이 아니라**,  
고객이 **유저 보이스를 계속 쌓고 → 회사별 Repository로 구조화하고 → 의사결정 순간에(로컬 에이전트 포함) 꺼내 쓸 수 있는 인프라**다.

### 1.2 제품 원칙 (반드시 갖춰야 할 모습)

| 원칙 | 전략/전략대화 근거 | 제품 의미 |
|------|---------------------|-----------|
| **정성조사 Moderation에 특화** | 포지셔닝·ROI·정량과 차별화 | 설문/폼 도구가 아니라 **AI 모더레이션 중심**의 인터뷰·FGI 워크플로우. Build → Moderate → Report가 한 흐름. |
| **데이터가 쌓이고 재사용된다** | Moat: 유저 보이스 데이터 레이크 | 프로젝트 단위로 끝나지 않고, **회사(또는 팀) 단위 Repository**에 발화·메타데이터가 누적. 태그·세그먼트·프로젝트로 구조화. |
| **에이전트·외부 툴과 연동 가능** | Moat: 로컬 에이전트 참조 | Repository를 **API·내보내기·연동**으로 노출해, Cursor 등 로컬 에이전트나 내부 위키/노션 등이 “유저가 뭐라고 했는지”를 참조할 수 있게 함. |
| **에이전시를 강화하는 도구** | 에이전시는 협력 채널, 대체 아님 | 멀티 클라이언트·멀티 프로젝트 지원. 에이전시가 **클라이언트별로 데이터를 구분**해 쌓고, 클라이언트에게는 “당신 회사 전용 Repository” 뷰 제공 가능. |
| **의사결정 순간에 쓰인다** | 의사결정 OS의 일부 | “회의 전에”, “PRD 쓸 때”, “전략 워크숍 전에” **유저 발화·인사이트를 검색·인용·요약**해서 쓸 수 있는 경로. 리포트만이 아닌 **질의·클립·요약** 경험. |
| **단순 “폼+대시보드”가 아니다** | 데이터 관리형 앱 소멸 논리 | 입력·저장·조회만 하는 UI가 아니라, **모더레이션 품질·한국어·패널·컴플라이언스** 등 에이전트가 쉽게 대체 못 하는 자산과 결합된 제품.

## 2. 페르소나별 제품 Use Case

### 2.1 유니콘 리서처 / 인사이트 팀 (Beachhead, Year 1)

| Use Case | 설명 | 성공 기준 |
|----------|------|-----------|
| **UC-1. 해외 시장 정성 리서치 단기 실행** | 일본·동남아 등 비한국어 시장에서 IDI/FGI가 필요할 때, 패널·리크루팅·동시통역 부담 없이 Proby로 프로젝트 설계 → AI 모더레이션(다국어) → 리포트까지 실행. | 2주 이내 설계부터 리포트 도출, 기존 대비 비용·일정 절반 이하 체감. |
| **UC-2. 프로젝트 결과를 팀/회사 Repository에 적재** | 한 프로젝트가 끝나면 “이번 프로젝트” 태그로 발화·요약이 **회사 Repository**에 자동(또는 원클릭) 적재. 나중에 “일본 사용자 반응” 검색 시 이 데이터가 포함되도록. | 리서처가 추가 작업 없이 또는 최소 클릭으로 Repository 적재 완료. |
| **UC-3. 의사결정 직전 “유저 목소리” 빠르게 참조** | 분기 전략 회의·로드맵 회의 전, “가격 정책 관련해서 유저가 뭐라고 했지?”를 Repository 검색(또는 에이전트 연동)으로 클립·요약 확인. | 5분 이내 관련 발화·요약을 찾아 1페이지나 슬랫에 인용 가능. |

## 3. 핵심 User Flow 시나리오

### 3.1 Flow A: “한 번의 정성 프로젝트를 끝내고 Repository에 남긴다”

**역할**: 유니콘 리서처 또는 대기업 인사이트 팀  
**목표**: 해외(또는 국내) 정성 조사 1건을 설계부터 리포트까지 수행하고, 결과를 회사 Repository에 쌓아 두기.

1. **프로젝트 생성**  
   - 목적·타깃(예: “일본 20대 사용자 제품 인지도”), 언어·참가자 수·예상 일정 입력.  
   - 템플릿 또는 가이드로 “정성 IDI/FGI” 타입 선택.

2. **참가자 확보**  
   - Proby 패널 연동 또는 자체 리크루팅 리스트 업로드.  
   - 초대 링크 발송, 참가 동의·스케줄 확인.

3. **질문·모더레이션 설계**  
   - 코어 질문·프로브·브랜치 설정.  
   - AI 모더레이터 옵션(언어, 톤, 재질문 규칙) 설정.  
   - (선택) 한국어 외 언어 지원 시 번역·로컬 맥락 반영 표시.

4. **인터뷰 실행**  
   - 참가자가 링크로 접속 → AI 모더레이터와 비대면 진행.  
   - 실시간 또는 사후 녹음·전사·구조화.

5. **리포트 생성**  
   - One-click 리포트(요약, 테마, 대표 발화, 시사점).  
   - 필요 시 수정·추가 섹션.

6. **Repository 적재 (제품 필수)**  
   - “이 프로젝트를 [회사/팀 Repository]에 저장” 액션.  
   - 메타데이터: 프로젝트명, 태그(예: 일본, 20대, 제품인지도), 날짜, 담당자.  
   - 이후 검색·에이전트 연동 시 이 데이터가 포함되도록.

**제품에 필요한 것**:  
- 끝단에 **“Repository에 저장”**이 기본 옵션인 워크플로우.  
- 프로젝트 ↔ Repository 매핑이 명확한 데이터 모델(회사/팀/태그).

---

### 3.2 Flow B: “의사결정 직전에 Repository에서 유저 목소리를 꺼낸다”

**역할**: PM 또는 리서처  
**목표**: 전략 회의·기획 리뷰 전에 “가격/UX/브랜드 관련 유저 발화”를 빠르게 찾아 1페이지나 슬라이드로 정리.

1. **Repository 접근**  
   - 대시보드에서 “우리 팀(또는 회사) 유저 보이스” 또는 “Repository” 진입.

2. **검색·필터**  
   - 키워드: “가격”, “비싸다”, “구독” 등.  
   - 필터: 기간, 프로젝트, 세그먼트(국가/연령/제품), 태그.

3. **결과 확인**  
   - 발화 클립 목록(인용 가능한 문장 단위 또는 문단 단위).  
   - 출처(프로젝트, 참가자 ID 마스킹), 날짜 표시.

4. **인용·공유**  
   - 클립 선택 → “1페이지 요약 생성” 또는 “클립 모음 export”.  
   - 내부 링크 또는 PDF/노션 등으로 전달.  
   - (장기) “이 내용을 Cursor/에이전트에게 참조시키기” 연결.

**제품에 필요한 것**:  
- Repository 전용 **검색·필터 UI**.  
- 클립 단위 인용·요약·export.  
- (Phase 2) API 또는 에이전트 연동으로 “에이전트가 Repository를 쿼리” 가능.

---

### 3.3 Flow C: “에이전시가 여러 클라이언트 프로젝트를 Proby로 돌리고, 클라이언트는 자기 Repository만 본다”

**역할**: 에이전시(실행), 인하우스(클라이언트, 소유)  
**목표**: 에이전시는 효율적으로 여러 프로젝트를 실행하고, 클라이언트는 자기 회사 데이터만 Repository로 받아 재사용·에이전트 연동까지 할 수 있게.

1. **에이전시: 클라이언트·프로젝트 생성**  
   - 클라이언트(예: A사, B사) 단위로 워크스페이스 또는 폴더 생성.  
   - 프로젝트 생성 시 “소속 클라이언트” 선택.  
   - 리크루팅·모더레이션·리포트는 Flow A와 동일.

2. **에이전시: 결과 배달**  
   - 프로젝트 종료 시 “A사 Repository에 배달” 액션.  
   - A사에게는 “당신 회사 Proby Repository에 새 프로젝트가 추가되었습니다” 알림(또는 메일).

3. **인하우스(A사): 자기 Repository만 보기**  
   - A사 계정으로 로그인 시 **A사 데이터만** 노출.  
   - 검색·필터·인용은 Flow B와 동일하되, 데이터 범위는 A사 한정.

4. **인하우스(A사): 에이전트 연동(선택)**  
   - A사가 API 키 또는 연동 설정을 하면, A사 로컬 에이전트가 **A사 Repository만** 참조.  
   - 기밀 유지: 에이전시·다른 클라이언트 데이터는 절대 노출되지 않음.

**제품에 필요한 것**:  
- **멀티 테넌시**: 클라이언트(또는 회사) 단위 데이터 격리.  
- 에이전시용 “클라이언트 관리” 화면과, 인하우스용 “내 회사 Repository” 화면 분리.  
- 권한: 에이전시 = 실행·배달, 인하우스 = 소유·조회·export·연동.

---

### 3.4 Flow D: “로컬 에이전트가 Proby Repository를 참조해 PM 질문에 답한다”

**역할**: PM(또는 디자이너), 로컬 에이전트(Cursor 등)  
**목표**: “이 기능에 대한 유저 반응 요약해줘”라고 하면, 에이전트가 Proby에 쌓인 유저 발화를 참조해 요약·인용.

1. **연동 설정(1회)**  
   - PM이 Proby에서 “에이전트 연동” 또는 “API 접근” 설정.  
   - 로컬 에이전트(예: Cursor 플러그인 또는 내부 봇)가 Proby Repository **읽기 전용**으로 접근할 수 있도록 토큰·엔드포인트 부여.

2. **일상 작업 중 질의**  
   - PM이 Cursor(또는 내부 채팅)에 “결제 플로우 개선 관련해서 우리 유저가 뭐라고 했는지 요약해줘” 입력.  
   - 에이전트가 Proby API로 해당 키워드/태그 검색 → 발화 클립·요약을 가져와 답변에 인용.

3. **출처 추적**  
   - 에이전트 답변에 “Proby 프로젝트: OOO, 2026-01” 등 출처가 포함되거나, 링크로 원본 발화 확인 가능.

**제품에 필요한 것**:  
- **Repository 읽기 API**(검색·필터·클립/요약 반환).  
- 보안: 회사 단위 인증, 클라이언트(회사)별 데이터만 노출.  
- (선택) Cursor·Slack 등 특정 에이전트용 플러그인 또는 봇.

## 4. 제품이 갖춰야 할 기능 요약 (전략·전략대화 기준)

### 4.1 현재 USP 유지 (필수)

- **High quality, human-like, zero-biased AI moderation**  
  - 한국어·다국어, 정성조사 맥락에 맞는 질문·프로브·재질문.
- **Panel Integration**  
  - 참가자 풀 연동 또는 업로드, 초대·스케줄 관리.
- **One-click report**  
  - 요약, 테마, 대표 발화, 시사점 자동 생성.
- **Affordable cost**  
  - 크레딧 기반 등 비용 구조가 정성 대비 ROI가 드러나도록.

### 4.2 Moat·전략대화 반영 (필수로 보강할 모습)

| 기능 영역 | 제품 모습 |
|-----------|-----------|
| **회사(팀) 단위 Repository** | 프로젝트가 끝나도 “한 번 쓰고 끝”이 아니라, **발화·메타데이터가 회사(또는 팀/클라이언트) 단위로 누적**되고, 태그·프로젝트·세그먼트로 검색 가능. |
| **Repository 검색·인용·공유** | “유저 보이스 검색” 전용 화면. 키워드·필터 → 클립 목록 → 1페이지 요약/클립 모음 생성, 내부 공유 링크 또는 export. |
| **에이전트·외부 연동** | Repository **읽기 API** (검색, 클립/요약 반환). 로컬 에이전트가 회사 데이터만 참조할 수 있도록 인증·스코프. (선택) Cursor·노션 등 플러그인. |
| **에이전시 멀티 클라이언트** | 에이전시가 **클라이언트별로 프로젝트 실행**하고, **클라이언트에게는 자기 회사 Repository만** 노출. 권한: 실행 vs 소유·재사용 분리. |
| **의사결정 순간 UX** | “회의 전”, “PRD 작성 시” 등 **타이밍에 맞는 진입점**: “이 주제로 유저 발화 보기”, “이 프로젝트 발화 1페이지로” 등 빠른 액션. |

### 4.3 피해야 할 모습 (전략대화 반영)

- **“폼 만들기 + 응답 모으기 + 대시보드”에 그치는 제품**  
  → 에이전트에 흡수될 위험이 크므로, **모더레이션 품질·패널·컴플라이언스·Repository** 등 자산과 결합된 형태 유지.
- **프로젝트 단위로만 데이터가 남고, 회사가 재사용하지 못하는 구조**  
  → Moat가 되려면 **회사별 누적·재조회·에이전트 연동**이 기본 시나리오여야 함.
- **에이전시를 “대체”하는 포지셔닝**  
  → 제품은 **에이전시가 클라이언트에게 더 잘 배달하고, 클라이언트는 자기 Repository를 가진다**는 “강화” 구조를 지원해야 함.

## 5. 제품 로드맵 관점 정리

| 단계 | 제품 초점 |
|------|-----------|
